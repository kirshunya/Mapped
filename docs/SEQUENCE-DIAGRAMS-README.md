# Диаграммы последовательностей (Sequence Diagrams) — Mapped

## 📋 Обзор

Диаграммы последовательностей показывают взаимодействие между 10 ключевыми компонентами системы в 5 основных сценариях использования проекта **Mapped**.

---

## 🏗️ 10 ключевых компонентов архитектуры

| № | Компонент | Описание | Технология |
|----|-----------|---------|-----------|
| 1 | **Frontend (React)** | Веб-интерфейс, локальное состояние, UI компоненты | React, TypeScript |
| 2 | **API Gateway** | Маршрутизация запросов, балансировка нагрузки | Gin/HTTP |
| 3 | **Auth Service** | Регистрация, логин, JWT токены, валидация | Go, bcrypt, JWT |
| 4 | **Posts Service** | Создание, чтение, редактирование постов, реакции | Go, Gin, GORM |
| 5 | **Places Service** | Поиск мест, создание, управление группами, лайки | Go, Gin, GORM |
| 6 | **Reviews Service** | Создание отзывов, рейтинги, комментарии | Go, Gin, GORM |
| 7 | **Chat Service** | Обмен сообщениями, групповые чаты, вложения | Go, Gin, GORM, WebSocket |
| 8 | **Media Service** | Загрузка файлов, преобразование, кеширование | Go, Gin |
| 9 | **Database (PostgreSQL)** | Персистентность данных всех сервисов | PostgreSQL, GORM ORM |
| 10 | **File Storage** | Хранилище медиа-файлов (локальное/S3) | Local FS / AWS S3 |

## 📊 5 основных сценариев использования

### 🎯 Сценарий 1: Создание поста с местом и медиа

**Участники:** User → Frontend → API Gateway → Posts Service → Places Service → Media Service → Database → File Storage

**Шаги:**
1. Пользователь открывает форму создания поста
2. Загружает изображение на сервер (Media Service)
3. Получает URL медиа
4. Заполняет содержание поста и выбирает место
5. Posts Service создаёт пост в БД
6. Places Service обновляет счётчик мест в посте
7. ✅ Пост создан успешно

**Файл:** `sequence-diagrams.mmd` (строки 11-67)

---

### ⭐ Сценарий 2: Лайк места и оставление отзыва

**Участники:** User → Frontend → API Gateway → Places Service → Reviews Service → Media Service → Database → File Storage

**Шаги:**
1. Пользователь открывает страницу места
2. Видит информацию (рейтинг, количество отзывов)
3. Кликает лайк на место → PlaceLike добавляется в БД
4. Загружает фото отзыва (Media Service)
5. Заполняет форму отзыва (рейтинг 0-5, текст, медиа)
6. Reviews Service создаёт отзыв и пересчитывает средний рейтинг места
7. Places Service обновляет Place (новый рейтинг, счётчик отзывов)
8. ✅ Отзыв опубликован, рейтинг места обновлён

**Файл:** `sequence-diagrams.mmd` (строки 69-155)

---

### 📍 Сценарий 3: Поиск мест рядом и просмотр деталей

**Участники:** User → Frontend → API Gateway → Places Service → Reviews Service → Database

**Шаги:**
1. Frontend получает GPS координаты пользователя
2. Отправляет запрос: GET /places/nearby?lat, lng, radius
3. Places Service выполняет geo-запрос к БД
4. Сортирует результаты по рейтингу
5. Возвращает 10 ближайших мест
6. Frontend показывает маркеры на карте
7. Пользователь кликает на место → видит детали и отзывы
8. Reviews Service возвращает все отзывы для места
9. Frontend регистрирует просмотр места (PlaceView)
10. ✅ Просмотр записан для аналитики

**Файл:** `sequence-diagrams.mmd` (строки 157-233)

---

### 🔐 Сценарий 4: Регистрация, логин и обновление профиля

**Участники:** User → Frontend → API Gateway → Auth Service → Database → Media Service → File Storage

**Шаги:**

#### Регистрация:
1. Пользователь вводит email, username, пароль
2. Auth Service валидирует данные
3. Проверяет, что email уникален
4. Хеширует пароль (bcrypt)
5. Сохраняет User в БД с role='user'
6. Генерирует JWT токен (sub: userID, role, exp)
7. Frontend сохраняет токен в localStorage
8. ✅ Регистрация завершена

#### Логин:
1. Пользователь вводит email и пароль
2. Auth Service найдёт User по email
3. Сравнивает пароль (bcrypt.CompareHashAndPassword)
4. Генерирует новый JWT токен
5. Frontend обновляет токен
6. ✅ Вход выполнен

#### Обновление профиля:
1. Пользователь открывает свой профиль
2. Frontend отправляет GET /auth/profile с Authorization: Bearer token
3. Auth Service валидирует токен и возвращает User
4. Пользователь загружает новый аватар
5. Media Service сохраняет файл и возвращает URL
6. Frontend отправляет PUT /auth/profile {avatar, bio}
7. Auth Service обновляет User в БД
8. ✅ Профиль обновлён

**Файл:** `sequence-diagrams.mmd` (строки 235-347)

### 💬 Сценарий 5: Создание группы и обмен сообщениями

**Участники:** User1 → Frontend1 → API Gateway → Places Service → Chat Service → Database ← Frontend2 ← User2

**Шаги:**

#### Создание группы:
1. User1 открывает форму создания группы
2. Вводит название и описание
3. Places Service создаёт Group в БД
4. Автоматически добавляет User1 как 'owner'
5. ✅ Группа создана

#### Добавление участника:
1. User1 кликает "Пригласить участников"
2. Вводит ID User2
3. Places Service проверяет права (owner может добавлять)
4. Добавляет GroupMember в БД
5. ✅ User2 добавлен в группу

#### Обмен сообщениями:
1. User1 открывает групповой чат
2. Chat Service загружает историю сообщений (последние 50)
3. User1 пишет сообщение с фото и отправляет
4. Chat Service сохраняет ChatMessage и MessageAttachment в БД
5. **WebSocket транслирует сообщение всем участникам группы**
6. Frontend2 получает новое сообщение через WebSocket
7. User2 видит сообщение от User1
8. User2 отправляет ответ
9. **WebSocket транслирует ответ User1**
10. ✅ Чат синхронизирован в реальном времени

**Файл:** `sequence-diagrams.mmd` (строки 349-428)

## 📈 Поток данных между компонентами

```
Frontend (React)
    ↓ HTTP/REST
API Gateway (Маршрутизация)
    ↓
    ├─→ Auth Service     ←→ Database
    ├─→ Posts Service    ←→ Database
    ├─→ Places Service   ←→ Database
    ├─→ Reviews Service  ←→ Database
    ├─→ Chat Service     ←→ Database
    └─→ Media Service    ←→ File Storage

Database (PostgreSQL)
    └─→ Все сервисы используют GORM ORM

File Storage (S3/Local)
    └─→ Media Service управляет загрузкой/удалением
```

## 🎓 Как читать диаграммы последовательностей

1. **Вертикальная ось** = время (сверху вниз)
2. **Горизонтальная ось** = участники (компоненты)
3. **Стрелки вниз** = сообщения/запросы
4. **Стрелки вверх (пунктир)** = ответы
5. **Числовые метки** = порядок взаимодействия
6. **Прямоугольники** = операции в компонентах
7. **Примечания** = логика обработки
